# Astro Wiki 重构设计

## 目标说明

将 USC Wiki 从 VitePress 文档站重构为基于 Astro 的校园 Wiki / Blog 混合型内容站。新站点继续把 `docs/` 作为内容仓库，保留接近 Obsidian 的 Markdown 写作习惯，并把文章之间的关系作为核心能力：分类、标签、双链、反向链接和相关页面都应由内容自动生成。

## 设计目标

- 继续围绕 `docs/` 维护 Markdown 内容。
- 使用 frontmatter 作为顶栏导航、分类聚合和 wiki/blog 组织方式的来源。
- 支持 Obsidian 风格链接：`[[页面]]`、`[[页面|别名]]` 和 `![[附件.png]]`。
- 根据内容元数据和链接关系生成分类页、标签页、反向链接和相关页面。
- 保留现有互动工具，例如 GPA 计算器、倒计时、Markdown 编辑器和资源卡片。
- 将视觉结构调整为更适合 wiki/blog 的形态：顶栏分类、内容列表、文章页和关系浏览。

## 非目标

- 不做完整的浏览器版 Obsidian，不包含在线编辑、同步或本地 vault 管理。
- 不强制把所有 Markdown 文件迁移到 Astro 默认的 `src/content` 目录。
- 第一阶段迁移时，未解析的双链不阻断生产构建。
- 不新增登录、评论、用户系统或动态服务端功能。

## 内容模型

`docs/` 下的 Markdown 文件继续作为主要内容来源。每篇公开文章最终应逐步补充如下 frontmatter：

```yaml
---
title: 高等数学 A1
category: 学业
tags: [课程, 数学, 期末]
description: 高数 A1 课程资料与复习入口
date: 2026-03-20
updated: 2026-05-21
draft: false
pinned: false
---
```

必填字段：

- `title`：页面显示标题。
- `category`：一个主分类，用于顶栏导航和分类页。

可选字段：

- `tags`：跨分类主题，用于标签页和相关内容推荐。
- `description`：列表页和预览区展示的摘要。
- `date`：原始发布或创建日期。
- `updated`：最近一次有意义的内容更新日期。
- `draft`：为 `true` 时在生产环境列表中隐藏。
- `pinned`：为 `true` 时在首页和分类页中优先展示。
- `order`：为少量需要人工排序的页面提供兜底排序字段。

如果迁移期页面缺少 `title`，构建过程应从第一个 H1 或文件名推导标题。如果缺少 `category`，第一阶段应根据当前一级目录使用受控映射补齐分类。

## 导航结构

顶栏导航由文章的 `category` 自动聚合生成，不再以文件夹作为唯一来源。初始分类沿用当前站点语义：

- 学业
- 生活
- 资源
- 工具

每个分类页展示该分类下的文章列表，包含标题、摘要、标签、更新时间和置顶状态。文件夹树可以作为辅助浏览方式存在，但不再是站点的主要信息架构。

## 路由设计

Astro 站点应提供以下路由：

- `/`：首页，展示置顶内容、最近更新和主要分类。
- `/notes/[...slug]`：由 `docs/**/*.md` 生成的文章详情页。
- `/categories/[category]`：分类索引页。
- `/tags/[tag]`：标签索引页。
- `/graph`：轻量关系图页面。
- `/tools/*`：从现有 VitePress/Vue 迁移过来的互动工具页面。

文章公开路径应尽量保留当前内容的逻辑 slug。第一阶段可以优先使用 `/notes/<docs 相对路径>` 来降低迁移风险，待链接覆盖情况验证后再补充重定向或别名。

## 链接关系系统

构建期内容索引应扫描所有 Markdown 文件，并生成以下信息：

- 文章元数据
- slug 和源文件路径
- 标题别名
- 出链
- 反向链接
- 未解析双链
- 附件引用
- 标签和分类

双链解析顺序：

1. 精确匹配 `title`。
2. 精确匹配文件 basename。
3. 精确匹配去掉扩展名的相对路径。
4. 在去掉 `.md` 后进行唯一规范化匹配。

如果链接目标存在歧义，应在开发诊断中标记为未解析，而不是静默选择某一个目标。

文章页应展示关系区域，包括反向链接、同标签相关页面和同分类相关页面。关系区域应保持有用且不过度打扰；没有内容的区块应自动隐藏。

## 附件处理

现有附件目录应继续可用：

- `docs/public`
- `docs/static`
- `docs/**/attachment`
- `docs/**/attachments`
- `docs/**/static`

`![[附件.png]]` 应优先按文件名解析，再结合引用页面附近的目录进行偏好选择。如果多个附件同名，构建过程应优先选择距离当前 Markdown 最近的 `attachment`、`attachments` 或 `static` 目录，并输出歧义警告。

## 布局与视觉方向

站点应像一个被持续维护的校园 Wiki，而不是营销落地页。桌面端文章页默认结构：

- 顶部：由分类生成的导航栏。
- 左侧：辅助浏览区，例如分类列表或文件树。
- 中间：文章正文。
- 右侧：页面目录和关系信息，在空间足够时显示。

移动端应将导航折叠为抽屉或紧凑菜单。反向链接和相关内容应移动到正文下方。

整体视觉应安静、易读、内容优先，同时具备现代学生维护知识库所需要的精致感。

## 组件迁移

现有 VitePress Vue 组件应迁移为 Astro islands 或等价组件：

- `GPACalculator`
- `Countdown`
- `HomeEnhancement`
- `LiveEditor`
- `VditorEditor`
- `ResourceCard`
- `ComingSoon`

当前在 Markdown 中直接嵌入 Vue 组件的页面，应尽量保留相近的写作方式。如果 Astro 中必须采用新的导入约定，迁移时只调整受影响页面。

## 迁移计划

第一阶段：Astro 基础

- 在现有项目中并行加入 Astro 配置。
- 配置 Markdown 渲染和 `docs/` 内容扫描。
- 建立内容索引，并验证 frontmatter fallback 行为。

第二阶段：Wiki 关系能力

- 实现双链和附件解析。
- 生成分类、标签、文章和关系图路由。
- 在文章页加入反向链接和相关内容。

第三阶段：现有功能对齐

- 迁移互动组件。
- 验证使用自定义 HTML 或 Vue 组件的页面。
- 保留数学公式、任务列表、脚注、高亮、代码高亮和资源卡片能力。

第四阶段：视觉系统

- 实现顶栏导航、文章布局、分类页、标签页和首页。
- 调整移动端导航和阅读体验。

第五阶段：切换上线

- 对比 Astro 生成路由和当前 VitePress 站点路由。
- 为重要旧 URL 添加重定向或别名。
- 在 Astro 构建和抽查通过后，再移除 VitePress 专属配置。

## 验证方式

每个实现阶段至少需要完成以下验证：

- 依赖变更后 `bun install` 成功。
- Astro 构建成功。
- 内容索引能输出分类、标签、双链、反向链接、未解析链接和附件引用。
- 代表性页面能正常渲染：
  - 首页
  - 一篇包含双链的学业文章
  - 一篇包含附件的资源页面
  - 一个包含互动组件的工具页面
  - 一个标签页
  - 一个分类页
- 使用浏览器抽查桌面端和移动端宽度。

## 风险

- 现有 Markdown 中包含 Vue 组件和原生 HTML，可能需要有针对性的迁移。
- 中文文件名和附件名需要谨慎处理 URL 编码和 slug。
- 多个文件或附件同名时，Obsidian 风格解析可能产生歧义。
- 保留 `docs/` 在 Astro 默认 `src/content` 之外，可能需要自定义 loader 或扫描工具。
- 最终切换前必须显式验证与现有 VitePress 路由的兼容性。
